## Minimum Viable Product (MVP)

### Definition

- Minimum Viable Product (MVP): The smallest version of a product that can solve a real user problem and generate meaningful learning.
- Viable: Must work well enough to solve a real problem.
- Minimum: Must exclude unnecessary features and “nice-to-haves.”

### Common Misunderstandings

- Cutting features randomly: Removing functionality without ensuring usefulness.
- Shipping broken products: Releasing incomplete or non-functional solutions.
- Learning from bad products: You cannot learn much beyond “people don’t like bad products.”

### What an MVP Should Achieve

- Validate problem-solution fit: Confirm the feature solves a real problem for a specific user.
- Encourage real usage: Test whether users _will_ use it (not just whether they _can_ use it).
- Generate actionable feedback: Reveal missing requirements and incorrect assumptions.

## Prototypes vs. MVPs

### Key Difference

- Prototypes: Test if users _can_ use a feature.
- MVPs: Test if users _will_ use a feature in real conditions.

### When MVPs Are Necessary

- Behavioral validation: When you need evidence of actual adoption.
- High uncertainty features: When the team is unsure which solution is best.
- Large investments: When building the full feature would be expensive and risky.

## Example: Job Marketplace Assessment Feature

### The Problem

- Employers struggle to assess whether job seekers can perform the job.

### The Risk

A full assessment system could include:

- Custom test creation
- Test reuse and editing
- Cross-posting assessments
- Candidate dashboards
- Anti-cheating measures
- Admin tools
- Customer support workflows

This becomes:

- Expensive
- Time-consuming
- Risky if wrong

### The Core Question

- Will employers actually use assessments?
- Are assessments the right solution?
- Would another solution (e.g., trusted referrals) work better?

## Finding the Core (MVP Exercise)

A collaborative technique to define what is truly required for an MVP.

### Step 1: Review Research

1. Identify user problems: What problem are we solving?
2. Define target users: Who is this for?
3. Consider risks: What harm might this feature cause?
4. Align on goals: Ensure shared understanding across the team.

### Step 2: Brainwrite (Writestorm)

- Silent idea generation (5 minutes).
- One requirement per sticky note.
- Cross-functional input:
  - Engineers: Architecture, security, infrastructure.
  - Customer support: Operational impact.
  - Product managers: Resourcing and constraints.
  - Designers: User workflows and experience.

### Step 3: Consolidate and Discuss

- Group duplicates.
- Clarify unclear items.
- Identify misalignment in understanding.
- Build a master list of all proposed requirements.

### Step 4: Define the Core

Draw two circles:

- Inner circle (Core): Requirements without which the feature would break.
- Outer circle: Enhancements and next-phase improvements.

#### Core Criteria

- Does the feature function at all without this?
- Does it solve the core user problem?
- Is it necessary to release and learn?

#### Example Distinction

- Allow job seekers to take assessments: Core (feature breaks without it).
- Reuse assessments across job postings: Enhancement (useful, but not required).

## Evaluating the MVP

After defining the core, ask:

1. Is this a complete, working product?
2. Does it solve a real problem?
3. Will releasing it help us learn?

If the answer is “no”:

- Add back only what is necessary to reach “yes.”

## Learning Outcomes from MVPs

- Identify which “requirements” are actually optional.
- Discover unexpected user preferences.
- Validate or invalidate the feature direction early.
- Avoid long investments in the wrong solution.

### Valid Outcomes

- Small feature works → Build on it.
- Small feature fails → Pivot or discard.
- Users want something slightly different → Adjust direction.

## Key Principles

- Not viable: If it doesn’t solve a real problem or produce learning.
- Not minimum: If filled with unnecessary extras.
- Agile goal: Reduce waste by building small, useful experiments.
- Strategic discard: Throwing away the wrong solution early is success.
