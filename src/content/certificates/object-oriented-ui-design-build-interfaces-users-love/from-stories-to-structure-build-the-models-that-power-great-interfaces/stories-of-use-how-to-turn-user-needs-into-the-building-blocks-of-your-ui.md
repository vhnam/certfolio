## Role of Stories of Use in the Design Process

### Stories of Use

- Stories of use: Narratives describing **how users achieve goals in a specific context**.
- Purpose: Connect **user research insights** to **design decisions**.
- Benefit: Help designers create **solutions grounded in real user needs**.

Key advantages:

- Clarify what matters to users.
- Provide structure for design thinking.
- Improve communication with team members.
- Support evidence-based design decisions.

### Stories of Use Across Product Development

Stories of use appear throughout the design lifecycle.

Stages:

1. **Research stage**
   - Collect stories from research participants.
   - Describe how users currently solve problems.
2. **Design stage**
   - Create design ideas.
   - Test those ideas using new stories of use.
3. **Model refinement**
   - Use stories to **expand and refine conceptual models**.

### Evolution of Conceptual Models

Conceptual models evolve through three levels.

- Essential models: Show **core concepts and relationships**.
- Specification models: Add **attributes, operations, and views**.
- Implementation models: Include **complete design details and prototypes**.

## Writing Stories of Use

### Starting with Problem-Oriented Stories

Early stories focus on **user goals in the problem domain**.

Examples:

- Record an activity (fitness app).
- Buy tickets to an event (entertainment platform).
- Cook a healthy meal with available ingredients (cooking app).
- Contact friends about a gathering (social app).

These early narratives are **problem stories**.

### Transition to Solution Stories

As designs progress:

- Stories begin using **concepts from the proposed solution**.
- They shift from **problem stories** to **solution stories**.

Validation methods:

- Card sorting
- Paper prototyping

Purpose:

- Ensure users understand the **terminology and structure of the design**.

### Updating Conceptual Models

After validating stories with users:

- Use them to **extend conceptual models**.
- Add new:
  - Views
  - Attributes
  - Operations

These additions become essential in **specification and implementation stages**.

## Identifying Stories from User Goals

### Hierarchical Goals

User goals are often **structured hierarchically**.

![A hierarchical goal structure.](../../../../assets/images/object-oriented-ui-design-build-interfaces-users-love/from-stories-to-structure-build-the-models-that-power-great-interfaces/object-oriented-ui-design-illustrations-user-goals.avif)

Levels:

- High-level goals
- Sub-goals
- Action-level tasks

Example goal hierarchy:

**Do a weekly grocery shop**

Sub-goals:

- Choose delivery date
- Add or remove items
- Checkout

Further breakdown:

**Checkout**

- Confirm shopping basket
- Enter delivery address
- Enter payment details
- Verify payment

### Epics and Stories

- Epic: A **large goal consisting of multiple sub-goals**.
- Stories of use: **Smaller action-focused tasks derived from epics**.

Development implication:

- Epics eventually split into **multiple user or persona stories** in the development backlog.

## Characteristics of Effective Stories of Use

### Focus on an Individual

- Stories should describe **one specific person**.
- This person can be:
  - A research participant
  - A persona representing a user group.

Reason:

- Designers empathize more easily with **individuals than groups**.

### Provide Context

- Stories should include **situational context**.
- Context clarifies **why the goal matters**.

Example situations:

- Need to complete a task quickly.
- Need to simplify a complex process.

### Describe Interaction-Level Actions

Stories used for model elaboration should focus on:

- **Specific user interactions with the system**.

Content includes:

- A sequence of user actions
- System responses
- Interaction steps

Higher-level stories may exist earlier in the process, but final stories should operate at the **interaction level**.

## Updating Conceptual Models from Stories

### Using Stories to Expand Models

Process for refining conceptual models:

1. Write or illustrate stories of use.
2. Review stories with team members.
3. Validate stories with potential users.
4. Walk through stories using the conceptual model.

### Model Expansion

While acting out the story:

- Add missing elements to the model.

Possible additions:

- New views
- New attributes
- New operations

Design support activities:

- Sketch interface ideas.
- Use visual aids to explore interactions.

## Key Takeaways

- Stories of use link **user research to practical design solutions**.
- They evolve from **problem stories** to **solution stories**.
- Large goals (epics) must be broken into **actionable stories** for development.
- Effective stories:
  - Focus on an individual user
  - Provide context
  - Describe interaction-level actions
- Validated stories help designers **refine conceptual models and guide implementation**.
