## Importance of Requirements in Design

### Definition of Requirements

- Requirements: Descriptions of what a system must do and the conditions it must satisfy.
- They form the **foundation of any design or development process**.

Good requirements consider:

- User needs
- Organizational goals
- Market competition
- Technical and regulatory constraints
- Platform limitations (e.g., app store rules)

When requirements are unclear:

- Design decisions become unstable.
- Development effort may be wasted.

### Role of Designers

Designers play a key role in shaping requirements by:

- Understanding user problems.
- Connecting business goals with user needs.
- Identifying constraints early in the process.

In user-centered design, the goal is to understand **problems from the user’s perspective** before designing solutions.

## Types of Requirements

Requirements are generally divided into two categories.

### Functional Requirements

- Functional requirements: Specific actions or behaviors that the system must perform.
- They describe **what the system should do**.

Examples:

- Users can create an account and log in.
- Customers can add items to a shopping basket and complete a purchase.
- Travelers can search for flights and reserve seats.
- Patients can book, reschedule, or cancel appointments.

Functional requirements can often be expressed as **stories of use**.

### Non-Functional Requirements

- Non-functional requirements: Constraints that shape how the system must operate.
- They define **conditions the system must satisfy**, not actions it performs.

Examples:

- Must run on Android and iOS.
- Must comply with financial security standards.
- Must be written in Swift.
- Must meet accessibility guidelines such as WCAG 2.x Level AA.

Important characteristics:

- They shape the design but do not define specific system behaviors.
- They can determine whether a product is legally or technically acceptable.

In many cases, non-functional requirements are **critical to project success**.

## Stories of Use

Functional requirements are often expressed as **stories of use**, which describe how people interact with a system.

Stories of use help designers:

- Understand user interactions
- Identify system behaviors
- Structure requirements around real usage

### Use Cases

- Use cases: Detailed descriptions of how a system responds to user actions.
- Often include:
  - Diagrams
  - Narrative descriptions
  - System responses

Use cases typically describe **complete interaction scenarios**.

### User Stories

- User stories: Short descriptions of functionality needed by users.

Typical structure:

1. Identify the user.
2. Describe the action.
3. Define the goal.

Example:

- A traveler wants to reserve a seat on a flight.

User stories are widely used in **Agile development**.

### Persona Stories

- Persona stories: User stories grounded in **real user research**.
- Based on **personas** rather than hypothetical users.

Benefits:

- More realistic
- Stronger connection to user needs
- Reduced reliance on assumptions

### Design Maps

- Design maps: Visual representations of user interactions.
- Typically created using:
  - Sticky notes
  - Digital equivalents

Design maps help teams:

- Visualize user flows
- Organize requirements
- Identify relationships between actions and objects

## Assumption-Based vs Research-Based Approaches

Two main approaches exist when writing stories of use.

### Assumption-Based Methods

These approaches imagine users rather than relying on real research.

Examples:

- Use cases
- User stories

Characteristics:

- Faster to create
- Based on assumptions about user behavior

### Research-Based Methods

These approaches rely on **actual user research**.

Examples:

- Persona stories
- Design maps

Characteristics:

- Grounded in real user behavior
- Better alignment with user needs
- Stronger support for user-centered design

## Challenges in Working with Requirements

Moving from requirements to design is difficult.

Challenges include:

- Requirements may be incomplete or ambiguous.
- Stakeholder priorities may conflict.
- Requirements often change over time.

Designers must:

1. Interpret and elaborate requirements.
2. Understand user goals and workflows.
3. Explore how users interact with the system.

This process transforms raw requirements into **usable design solutions**.

## Key Takeaways

- Requirements are the **starting point of all design work**.
- They define both **what the system does** and **the constraints it must satisfy**.
- Functional requirements describe system behaviors.
- Non-functional requirements define constraints such as platforms, security, and accessibility.
- Functional requirements can be expressed through **stories of use**.
- Tools like **use cases, user stories, persona stories, and design maps** help structure requirements.
- Research-based approaches produce **more reliable user-centered designs**.
- Clearly defined requirements reduce risks, align teams, and create stronger foundations for successful products.
