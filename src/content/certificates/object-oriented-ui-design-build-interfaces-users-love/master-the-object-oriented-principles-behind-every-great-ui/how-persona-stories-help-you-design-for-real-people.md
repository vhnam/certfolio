title: "How Persona Stories Help You Design for Real People"
---

## Problems with Traditional User Stories

### Definition

- User stories: Short requirement descriptions typically written as **“As a role, I want a goal so that a benefit.”**

Example:

- As a customer, I want to see a list of available flights so that I can make a booking.

### Limitations from a UX Perspective

User stories have several issues in **user-centered design**:

- Wrong focus: They emphasize **roles instead of real behaviors and needs**.
- Structural flaws: Often **wordy and written in first person**.
- Authorship: Frequently written **only by software developers**.
- Assumption-based: Based on **imagined users rather than research**.
- Premature creation: Often written **before real user needs are understood**.

Because of these problems, user stories often fail to provide meaningful insight into **who the user actually is**.

## Personas: Representing Real Users

### Definition

- Personas: Fictional characters that represent **groups of users with shared behaviors, goals, and needs**, created from **user research**.

Personas typically include:

- Name
- Photo
- Relevant demographics
- Goals and motivations
- Behavioral insights

### Purpose of Personas

Personas help teams:

- Design for **a concrete individual instead of an abstract user**.
- Build **empathy for users’ real experiences**.
- Maintain a **shared understanding of target users** across the team.

When personas are used consistently:

- Design decisions stay grounded in **real user needs**.
- Teams align around **who they are building for**.

## Persona Stories

### Definition

- Persona stories: Stories of use based on **research-backed personas rather than abstract roles**.

Unlike user stories, they are written in **third person** and refer to a **specific persona**.

Structure:

- Persona → Action → Goal

Example transformation:

User story:

- As a customer, I want to see a list of available flights so that I can make a booking.

Persona story:

- Bob views available flights to make a booking.

### Characteristics of Persona Stories

Persona stories are:

- Based on **researched user behaviors and needs**.
- Written **in the third person**.
- Created **collaboratively between UX practitioners and developers**.
- Written **after user research and early conceptual modeling**.

This approach ensures stories are grounded in **real user understanding** rather than speculation.

## Roles vs Personas

Roles do not disappear entirely when using persona stories.

Instead:

- Roles become **secondary context**.
- The persona remains the **central focus**.

Example:

- Bob may act as both **customer** and **passenger** in a booking system.
- However, he is always **Bob**, representing a real set of user behaviors.

This shift helps teams focus on **the person behind the role**.

## Benefits of Persona Stories

Persona stories improve requirements by providing:

- Empathy: Teams see problems from the **user’s perspective**.
- Evidence: Stories are grounded in **user research**.
- Clarity: Real user context improves decision-making.
- Alignment: Teams share a **common understanding of users**.

They change the design question from:

- “What do we think users want?”

to:

- “What do we know this person needs?”

## Elaborating Stories of Use

Both **user stories and persona stories** are initially **placeholders**.

Before implementation, they must be elaborated.

Elaboration helps:

- Clarify assumptions
- Identify edge cases
- Add necessary interaction details
- Prevent misunderstandings during development

This process should ideally involve:

- UX designers
- Developers
- Other project stakeholders

## Key Takeaways

- User stories evolved from use cases but often rely on **abstract roles and assumptions**.
- Personas represent **research-based user groups with shared behaviors and needs**.
- Persona stories replace roles with **specific researched personas**.
- They are written in **third person** and emphasize **real user context**.
- Persona stories should be created **after user research and early conceptual modeling**.
- Elaborating stories collaboratively ensures they become **actionable requirements for design and development**.
