## Understanding Epics

### Definition of Epics

- Epic: A **large user story** that contains multiple smaller stories and requires **more than one development cycle** to complete.
- Characteristic: Often appears simple initially but reveals **significant complexity during implementation**.
- Purpose: Helps organize complex functionality into **manageable units of work**.

Epics must be broken down into smaller stories so each one can be implemented within a **single sprint** in Agile development.

### Why Epics Must Be Split

Key reasons for splitting epics:

- Maintain **consistent implementation effort** across user stories.
- Avoid **scheduling issues during development sprints**.
- Improve clarity about **what work must be done**.

Example epic:

- Epic: Jane checks out of the hotel.

Possible child stories:

- Jane reviews charges.
- Jane confirms payment details.
- Jane requests an invoice.

Each smaller story becomes **individually implementable**.

## Identifying Epics in User Stories

### Hidden Complexity in Stories

Short user stories can sometimes hide **multiple underlying interactions**.

- User stories: Short descriptions of user interaction.
- Persona stories: Similar format but focused on a persona’s context.
- Use cases: More detailed narratives that often reveal complexity earlier.

Problem:

- A simple story may actually represent **multiple workflows or interactions**.

This hidden complexity is what defines an **epic**.

## Splitting Epics into Smaller Stories

### Step-Based Breakdown

Users naturally understand tasks as **sequences of steps**, which makes epics easier to divide.

Example: E-commerce checkout epic

1. Harry adds an item to the basket.
2. Harry reviews the basket contents.
3. Harry enters delivery and billing information.
4. Harry selects a payment method.

Each step becomes a **separate user story**.

### Complex Epic Example

Some epics require deeper analysis to split correctly.

Example epic: Manage subscriptions in a streaming service.

Possible stories:

- Renew or upgrade a subscription plan.
- Pause a subscription.
- Change payment method.
- Manage family or shared accounts.
- View billing history.

Although users perceive this as **one task**, it actually involves **multiple workflows and system interactions**.

## Collaboration When Breaking Down Epics

### Cross-Team Analysis

Epics should not be divided by a single role.

- UX designers: Understand **user needs and usability considerations**.
- Developers: Understand **technical feasibility and system complexity**.

Effective epic breakdown requires **collaboration between both groups**.

### Maintaining Epic Context

Best practice:

- Label child stories with the **name of their parent epic**.

Benefits:

- Keeps the relationship between stories clear.
- Provides context for why each story exists.
- Prevents confusion when reviewing individual stories.

## Key Takeaways

### Epics Reveal Hidden Complexity

Epics often appear as **simple user stories**, but they may contain multiple related tasks or workflows.

Recognizing them early prevents:

- Planning confusion
- Sprint scheduling problems
- Underestimated development effort

### Breaking Down Epics Improves Development

Effective epic management requires:

- Splitting large stories into **clear, implementable user stories**
- Ensuring each story **fits within a sprint**
- Maintaining **traceability to the parent epic**
- Collaborating between **design and development teams**

This process turns **large complex problems** into **coordinated, manageable work units**.
